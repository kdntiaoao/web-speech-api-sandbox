# Conventional Commit メッセージ生成（日本語）

ステージ済みの変更を 1 つの論理変更として読み解き、**Conventional Commits 1.0.0** に準拠した日本語のコミットメッセージを生成する。最終的な出力はコミットメッセージそのものだけにする(後述の「複数の論理変更を検出した場合」を除く)。前置きや解説、「以下が提案です」のような枕詞は付けない — ユーザーはそのまま `git commit` に貼れる状態を期待している。

## なぜこの手順なのか

良いコミットメッセージは「何を変えたか」だけでなく「なぜ変えたか」を後から読む人に伝える。diff を機械的に要約しただけのメッセージ(例: `fix: App.tsx を修正`)は、diff を見れば分かることの繰り返しで価値が薄い。だからこの skill は、まず変更の **意図** を理解し、リポジトリの既存の書き方に合わせ、必要なときだけ理由・影響を本文で補う、という順序を踏む。

## 手順

### 1. ステージ済みの変更を読む

```bash
git diff --cached --stat   # 変更ファイルの全体像
git diff --cached          # 実際の差分
```

`git diff --cached` が空なら、ステージされた変更が無い。その場合は「ステージ済みの変更がありません。`git add` でステージしてから再度実行してください」とだけ伝えて止まる(メッセージを捏造しない)。

差分からは「ファイルが変わった」ではなく「何の意図でこの変更が入ったか」を読み取る。新規ファイルなら何を追加したのか、修正なら何の問題を解いたのか、設定変更なら何を有効化したのかを掴む。

### 2. リポジトリの慣習を調べる

メッセージの type・scope・言い回しは、リポジトリ既存のスタイルに合わせるのが最優先。新しい流儀を持ち込まない。

```bash
git log --oneline -30      # 実際に使われている type / scope / 日本語の書き口
```

ここから次を読み取る:

- **使われている type**: `feat` `fix` `chore` `docs` `refactor` `style` `test` `perf` `build` `ci` など。標準外の type(例: `update`)が慣習化していれば、それを尊重する。
- **scope の有無と形式**: 既存コミットが scope を使っていなければ、無理に付けない。使っているなら同じ粒度(コンポーネント名・領域名など)に合わせる。
- **description の言い回し**: 体言止め(「〜追加」「〜修正」)か、ですます調か。文末の句点の有無。日本語か英語か。

git log から判断できない・履歴が浅い場合のみ、Conventional Commits 標準の type を fallback として使う。

### 3. 単一の論理変更か判定する

ステージ済みの変更が **1 つのまとまった意図** を表しているかを評価する。複数の無関係な変更(例: 「バグ修正」と「新機能追加」が混在、あるいは「機能実装」と「無関係なリファクタ」が同居)が混ざっている場合は、1 つのコミットに押し込めると履歴が読みづらくなり、revert もしづらい。

- **単一だと判断できる場合** → そのまま 4 へ進む。
- **明らかに複数の論理変更が混在する場合** → メッセージを 1 つに無理やりまとめず、まず簡潔に警告する。どの変更群に分けられそうか(例: 「(1) 認証ロジックの修正 / (2) README の更新」)を挙げ、`git add -p` などで分割コミットする選択肢を提示する。そのうえで「それでも 1 つにまとめる場合のメッセージ」を提示してよい。ここだけは「メッセージのみ」の原則の例外で、ユーザーが履歴の質を判断できるよう情報を添える。

判定は厳しすぎないこと。関連する変更(機能本体とそのテスト、実装とそれに付随する型定義など)は 1 つの論理変更として扱ってよい。

### 4. メッセージを組み立てる

次の構造で出力する:

```text
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

**タイトル行(必須)**

- `<type>`: 手順 2 で掴んだ慣習に従う。
- `[optional scope]`: 慣習が scope を使うときだけ、`feat(editor):` のように付ける。
- `<description>`: 日本語で簡潔に。**何をしたか** が一目で分かる体言止めや短い文にする。diff の単純な言い換えではなく、変更の主眼を書く。タイトル行は概ね 50 字以内を目安に短く保つ。

**本文(必要なときだけ)**

body は **理由・影響・主要な変更点** を説明する場所。付けるのは、それがタイトルだけでは伝わらない価値を持つときに限る:

- なぜこの変更が必要だったのか(背景・きっかけ)
- どんな影響・トレードオフがあるか(挙動の変化、互換性、既知の制約)
- 一目で追いにくい複数の変更点の要約

逆に、タイトルで意図が十分伝わる自明な小変更(設定追加、フォーマット、軽微な文言修正など)には body を付けない。空の body で行数を稼がない。body を付けるときはタイトルとの間に空行を 1 行入れる。

body を書くときは、流れる文章の段落より **markdown の箇条書き(`- `)** を優先する。後から履歴を追う人は body を「読む」より「スキャンする」ことが多く、理由・影響・変更点が箇条書きで分かれている方が一目で掴める。このリポジトリの既存コミットも body に箇条書きを使っているなら、その粒度に合わせる。理由が 1 点だけの短い body なら 1 行(または 1 つの箇条書き)で十分。各行は折り返しが必要なほど長くしない。

**フッタ(該当するときだけ)**

- **破壊的変更**: `BREAKING CHANGE: <説明>` をフッタに入れる。または type の直後に `!` を付ける(例: `feat!:`)。後方互換を壊す変更では必ず示す — これを読み落とすと利用側が壊れる。
- Issue 参照など、リポジトリの慣習にフッタがあればそれに合わせる(例: `Refs: #123`)。無ければ付けない。

### 5. 出力する

組み立てたメッセージ **だけ** を出力する。コードブロックで囲んでそのままコピーできる形にする。判定や慣習についての説明は付けない(手順 3 で複数変更を検出したケースを除く)。

## 例

**例 1 — 自明な小変更、body 不要**

ステージ内容: `.oxlintrc.json` に lint ルールを 1 つ追加。
出力:

```text
chore: oxlint に no-console ルールを追加
```

**例 2 — 理由・影響を伝える価値がある変更**

ステージ内容: 長文読み上げが途中で止まる問題に対し、句読点でフレーズ分割して逐次再生するよう変更。
出力:

```text
fix: 長文読み上げが途中で停止する問題を修正

- ブラウザの SpeechSynthesis には 1 回の発話長に上限があり、長文を一括で渡すと途中で停止していた
- テキストを句読点で分割し、各フレーズの onend で次を再生する逐次方式に変更して回避する
```

**例 3 — scope を使う慣習のリポジトリ**

ステージ内容: 認証フォームに JWT ベースのログインを実装。既存コミットが `feat(auth):` 形式の scope を使用。
出力:

```text
feat(auth): JWT ベースのログイン認証を実装
```

**例 4 — 破壊的変更**

ステージ内容: 公開 API のレスポンス形式を変更し後方互換を破壊。
出力:

```text
feat(api)!: レスポンス形式をキャメルケースに統一

BREAKING CHANGE: API レスポンスのキーがスネークケースからキャメルケースに変わる。既存のクライアントは参照キーの更新が必要。
```

**例 5 — 複数の論理変更を検出(例外的に警告を添える)**

ステージ内容: 読み上げのバグ修正と、無関係な README の大幅更新が混在。
出力:

> ⚠️ ステージ済みの変更に複数の論理変更が混在しています:
> 1. 読み上げ停止バグの修正(`src/App.tsx`)
> 2. README の更新(`README.md`)
>
> 履歴を読みやすく保つため、`git add -p` 等で分割コミットすることを推奨します。それぞれのメッセージ案:
>
> ```text
> fix: 長文読み上げが途中で停止する問題を修正
> ```
> ```text
> docs: README にセットアップ手順を追記
> ```
>
> それでも 1 つにまとめる場合:
> ```text
> fix: 読み上げ停止の修正と README 更新
> ```
